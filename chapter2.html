Prompt Engineering
Chapter 2 is the most hands-on chapter. The goal is to move the audience from "typing a question" to engineering precise, reproducible prompts. We'll dissect prompt anatomy, cover six proven patterns (Zero-Shot, Few-Shot, Chain-of-Thought, Role, System, Retrieval-Augmented), and end with an interactive playground. Encourage the audience to open ChatGPT alongside the slides and try the examples in real time.
Slide 01 · Anatomy

Anatomy of a Prompt

SYSTEM: You are a helpful travel planner. Be friendly and concise. ① Role

USER:
[Context] I'm planning a 5-day trip to Japan in April with my partner. Budget: ~€3,000 total. ② Context
[Task] Create a day-by-day itinerary with must-see places, local food spots, and transport tips. ③ Task
[Format] Use a table: Day | Morning | Afternoon | Evening | Est. Cost. ④ Format
[Constraints] Avoid tourist traps. Include at least one hidden gem per day. ④
  1. 1
    Role / System
    Who is the model? Sets tone, expertise, personality.
  2. 2
    Context
    Background info: who, what, when, budget, constraints.
  3. 3
    Task / Instruction
    What do you want? Be specific and action-oriented.
  4. 4
    Format & Constraints
    Output shape (table, list, JSON), length, rules to follow.
Walk through each layer top-to-bottom. The system prompt is the trust anchor — everything about the model's role, constraints, and persona. Role assignment ('You are a senior product manager...') measurably improves output quality. Output format instructions (JSON, bullet points, markdown) must be explicit — the model doesn't know your downstream requirements. Context window = everything the model can 'see' at once.

Slide 02 · Patterns

Prompting Patterns

Zero-Shot

No examples given. Direct instruction.

Classify this review as positive or negative:
"The food was amazing."

✓ Best for: simple, well-known tasks

Few-Shot

Provide 2-5 examples to guide the model.

"Great!" → Positive
"Terrible." → Negative
"OK." → Neutral

"The food was amazing." → ?

✓ Best for: custom formats, niche tasks

Chain-of-Thought

Ask the model to reason step by step.

Solve this. Think step by step.

Step 1: identify the variables…
Step 2: apply the formula…

✓ Best for: math, logic, complex reasoning

Rule of thumb: Start with zero-shot. If the output isn't right, add examples (few-shot). For reasoning tasks, always use chain-of-thought.
Rule of thumb: start with zero-shot (GPT-4/Claude handle most tasks well). If the format is wrong, switch to few-shot (2–5 examples). For math/logic/multi-step reasoning, always use chain-of-thought — either explicitly ('Let’s think step by step') or via a worked example in the prompt. These patterns stack: zero-shot + system prompt + CoT is a common combination.

Slide 03 · Advanced

Advanced Techniques

System Prompts
Hidden instructions that define the model's persona, rules, and guardrails. Users never see them.
SYSTEM: You are a helpful assistant for Acme Corp. Never discuss competitors. Always respond in JSON format.
Role Playing
Assign a specific expert role to improve output quality. The model adapts vocabulary, depth, and style.
Act as a cybersecurity auditor with 15 years of experience…
Output Formatting
Specify exactly what structure you want: JSON, Markdown table, bullet points, YAML, XML…
Return a JSON object with keys: title, summary, score (1-10), tags[]
Meta-Prompting
Ask the model to improve its own prompt. Iterate for better results.
Rewrite this prompt to be clearer, more specific, and get better results…
RAG (Chapter 4) and Tool Use (Chapter 5) are the real power multipliers. ReAct loops the model through Reason → Act → Observe cycles, enabling step-by-step API calls. Least-to-most prompting works well for curriculum-style decomposition. Tree of Thought is expensive but valuable for complex decision-making tasks. These techniques are for architects and senior devs — flag as 'advanced' for the audience.

Slide 04 · Best Practices

Do's & Don'ts

✓ Do
  • Be specific and explicit
  • Provide context and examples
  • Specify output format (JSON, table…)
  • Break complex tasks into steps
  • Iterate and refine your prompt
✗ Don't
  • Use vague instructions ("do something good")
  • Send multiple unrelated tasks at once
  • Assume the model remembers past chats
  • Trust output blindly (always verify!)
  • Share sensitive data in prompts
The #1 mistake: vague prompts. 'Summarize this' → 'Summarize this in 3 bullet points for a non-technical VP. Use plain English, no jargon.' Negative instructions ('don’t do X') are weak — replace with positive constraints ('always do Y'). Test on 10–20 diverse examples before shipping. Live demo: take a bad prompt from the audience and improve it together.

Slide 05 · In Practice

Real-World Examples

For Managers
Summarize this 20-page report in 5 bullet points. Focus on budget impact and key risks.

→ Zero-shot, format constraint

For Architects
Act as a cloud architect. Compare 3 approaches for real-time order processing: Kafka vs SQS vs EventBridge. Markdown table.

→ Role + format + specificity

For Developers
Review this Python function for: 1) bugs 2) performance 3) security. Show fixes. Think step by step.

→ CoT + structured output

Remember: Prompt engineering is iterative. Your first prompt is never your best. Refine, test, and improve.
Walk through 1–2 examples live — invite the audience to critique the 'before' prompt. The rewrite pattern is consistent: add role, add format specification, add context/constraints. Offer the prompt template library as a follow-up resource. Q&A opportunity: 'What use case are you trying to tackle? Let’s prompt-engineer it together.'

Slide 07 · Exercise

🛠️ Hands-On Exercise

Challenge: Prompt Refinement

Start with the naive prompt below. Iteratively improve it using patterns from this chapter until the output quality is excellent.

❌ NAIVE PROMPT
"Write me an email about a project delay"
✅ YOUR GOAL
Produce a professional, empathetic email that a PM could actually send — with specific details and next steps.
⚡ Steps to Follow
1. Add a Role — "You are a senior project manager…"
2. Add Context — project name, 2-week delay, reason, stakeholders
3. Specify Format — tone, length, structure (greeting, body, next steps, closing)
4. Add a Few-shot example of a good delay email
5. Use Chain-of-Thought — "Think step by step about what the recipient needs to know"
Give the audience 5-7 minutes to work on this individually or in pairs. They should use ChatGPT, Claude, or any available LLM. After the exercise, ask 2-3 volunteers to share their prompts and compare results. Key teaching moment: show how each refinement step dramatically improves quality. The naive prompt generates generic, unusable text. The engineered prompt produces something a PM could actually hit "send" on. This exercise reinforces all five patterns from the chapter.
