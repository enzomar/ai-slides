AI Agents + MCP
Chapter 5 is about the frontier — moving from chat to action. AI agents can plan multi-step tasks, call APIs, read databases, and execute code autonomously. We cover the ReAct loop (Reason + Act), tool use, the new MCP protocol from Anthropic, and Andrew Ng's four agentic design patterns. Key warning: agents are powerful but require robust guardrails — every tool call is a potential security surface.
Slide 01 · Concept

What is an AI Agent?

An AI agent is an LLM that can observe its environment, reason, decide, and take actions to reach a goal — in a loop.

LLM vs Agent
Plain LLM
Q → A (one shot)
AI Agent
Goal → Plan → Tools → Result
  1. 1
    Perceive
    Reads input: user message, tool results, memory
  2. 2
    Reason
    Decides what action (or tool) to use next
  3. 3
    Act
    Calls a tool: web search, code exec, API call, file write…
  4. 4
    Observe & Loop
    Checks result, repeats until goal is achieved
The key difference: a chatbot responds; an agent acts. Agents can take minutes or hours to complete a task, making multiple decisions along the way. The loop is often called Observe-Orient-Decide-Act (OODA) or in LLM terms, ReAct (Reason + Act). Risk: agents can get stuck in loops or take unintended actions — human oversight is critical. Current state (2025): most production agents are semi-autonomous with human-in-the-loop checkpoints. Andrew Ng (DeepLearning.AI) identifies 4 core agentic patterns: Reflection, Tool Use, Planning, and Multi-Agent Collaboration.

Slide 01b · Agentic Patterns (DeepLearning.AI)

Andrew Ng's 4
Agentic Design Patterns

🪞 Reflection
The agent reviews its own output, critiques it, and iterates. Like a writer editing their own draft before submitting.
Example: Code generation → self-review → fix bugs → re-test
🔧 Tool Use
The agent calls external tools — web search, code execution, APIs — to gather information and perform actions beyond text generation.
Example: Research question → search web → read papers → synthesize
📋 Planning
The agent creates an execution plan before acting: breaks complex goals into sub-tasks, orders them, and tracks progress.
Example: "Build a website" → plan → design → code → test → deploy
🤝 Multi-Agent Collaboration
Multiple specialised agents work together: one writes code, another reviews it, a third runs tests. Debate and consensus yield better results.
Example: Writer agent + Critic agent + Editor agent → polished article
Source: Andrew Ng, DeepLearning.AI — "Agentic Design Patterns" series, 2024. These patterns can be combined: an agent can plan, use tools, reflect on results, and collaborate with other agents.
Andrew Ng popularized these four patterns in his DeepLearning.AI newsletter and courses in 2024. The key insight: agentic workflows where the LLM iterates on its own output dramatically outperform zero-shot approaches. Reflection alone can boost code generation from ~50% to ~95% correctness. GPT-3.5 with agentic workflows often outperforms GPT-4 with simple prompting. Ask the audience: which pattern resonates most with your use case?

Slide 02 · ReAct

Reason + Act:
The ReAct Loop

Example: "Book me Paris flights for next Tuesday"

Thought: I need to find today's date first.
Action: get_current_date()
Observation: 2026-03-06

Thought: Next Tuesday is 2026-03-10. Search flights.
Action: search_flights("Paris", "2026-03-10")
Observation: 3 results found, cheapest €189

Thought: I should confirm before booking.
Action: ask_user("Confirm cheapest at €189?")
Observation: "Yes, confirm."

Action: book_flight(flight_id="AF1234")
✓ Booking confirmed. PNR: XK7291
🧠 Thought — Internal reasoning step
→
⚡ Action — Call a tool / function
→
👁️ Observation — Tool result returned
→
🔁 Repeat until goal reached or human asked
ReAct (Reason + Act) is the foundational pattern described by Yao et al. (2022). The model's internal thought process is laid out explicitly in the prompt — this 'chain-of-thought scratchpad' significantly improves task completion. Key risk: the loop can fail if a tool call fails or returns unexpected data — good agents have error handling and retry logic at each step.

Slide 03 · Tool Use

Tool Use &
Function Calling

You describe tools as JSON schemas. The model decides when and how to call them.

// Tool definition (schema)
{
  "name": "get_weather",
  "description": "Get current weather",
  "parameters": {
    "city": { "type": "string" }
  }
}

// Model output (function call)
{ "tool": "get_weather",
  "args": { "city": "Paris" } }
Common Tool Categories
  • 🔍 Search — web, vector DB, SQL queries
  • 💻 Code execution — Python sandbox, shell
  • 🌐 APIs — weather, maps, CRM, ERP, Slack
  • 📂 File system — read, write, create, delete
  • 📧 Communication — send email, calendar, Slack
  • 🧪 Testing — run tests, deploy, CI/CD triggers
Function calling (OpenAI) / tool use (Anthropic) is now native in all major models. The model emits a structured JSON payload instead of plain text. Your code executes the function and returns the result back to the model. This prevents arbitrary code injection. Safety principle: the agent requests; your code decides whether to execute (human-in-the-loop gate).

Slide 04 · MCP

Model Context Protocol
(MCP)

MCP — Anthropic, Nov 2024

An open protocol that standardises how AI applications connect to external tools, data sources, and services — like USB-C for AI integrations.

🤖 AI App
Claude, GPT
⇌
📡 MCP
Standard protocol
⇌
🔧 MCP Server
GitHub, Slack…
Available MCP Servers (2025)
  • GitHub — repos, PRs, issues, commits
  • Slack — read channels, send messages
  • PostgreSQL / SQLite — query databases
  • Google Drive / Docs — read & write files
  • Brave Search — real-time web search
  • Docker — manage containers
  • Jira / Linear — project management
  • + 1000s of community servers
MCP launched November 2024 and was quickly adopted by Cursor, VS Code Copilot, Windsurf, and most major AI development tools. Before MCP, every AI product had to build custom integrations for every tool. MCP creates a universal adapter layer — write the server once, works with any MCP-compatible AI. Analogy: like the Language Server Protocol (LSP) standardised IDE features across editors, MCP standardises AI tool integrations.

Slide 05 · Multi-Agent

Multi-Agent
Systems

Complex tasks can be split across multiple specialised agents that collaborate, verify each other, and hand off work.

Orchestrator ↔ Worker pattern

A "manager" agent breaks down the goal, delegates to specialised "worker" agents, collects results, and synthesises the final answer.

Real-World Agents (2025)
  • Devin — autonomous software engineer (Cognition)
  • SWE-agent — GitHub issue resolver (Princeton)
  • Claude Computer Use — controls desktop GUI
  • AutoGen / LangGraph — multi-agent frameworks
2025 trend: Agents are moving from demos to production — with careful guardrails.
Multi-agent systems shine for tasks too large for one context window, tasks requiring diverse skills, or tasks needing checks and balances (one agent writes, another reviews). Challenges: higher latency, cost, error propagation, and debugging difficulty. Best practice (2025): build the simplest agent that works. Add complexity only when single-agent fails. Always include a human-checkpoint for irreversible actions (send email, delete file, deploy to prod).
