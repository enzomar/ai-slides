Introduction

Imagine teaching a machine to predict the next word in a sentence — then scaling that up 100 billion times. Suddenly it writes code, translates poetry, and reasons about the world. This chapter unpacks how that miracle works.

Chapter 01 · Introduction

Deterministic
vs Probabilistic

Deterministic
Same input → Same output. Always.
0 1
  • SQL query → always same rows
  • SHA-256 hash function
  • 2 + 2 = 4 (every time)
  • Sorting algorithm
Probabilistic
Same input → Different possible outputs.
  • Coin flip — Heads 50% · Tails 50%
  • Dice roll — each face ~16.7% probability
  • Weather forecast — 70% chance of rain
  • Stock price tomorrow — distribution of possible moves
Key takeaway: LLMs are probabilistic — the same question can produce different, valid answers. This is a feature, not a bug.

Next-Token Prediction

LLMs generate text one token at a time — each step samples from a probability distribution over the entire vocabulary.

The Transformer Architecture

Core capability
Predict the
next token
…with context-aware attention

Modern AI is essentially this exact pipeline, scaled up to trillions of parameters.

GPT-2 · 1.5B LLaMA 3 · 70B GPT-4 · ~1.8T Claude · ?
End-to-end token flow
INPUT TOKENS
“The cat sat on the”
Context window: the model reads the tokens already present before predicting what comes next.
↓
Embedding
Converts token IDs into dense vectors so the model can operate on meaning and similarity, not raw text IDs.
↓
Self-Attention
Each token looks at the others and assigns weights: this is where the model decides which past words matter most right now.
↓
Feed-Forward
Transforms each token representation independently with learned non-linear layers — much of the model's stored knowledge lives here.
↓
Softmax
Turns the final scores (logits) into a probability distribution over the whole vocabulary.
↓
Sampling
Applies temperature, top-k, or top-p rules and actually picks one candidate token from the probability distribution.
↓
OUTPUT TOKEN
“mat”
The chosen token is appended to the sequence, becomes part of the new context, and the whole loop runs again.

Key Parameters

Temperature
Controls randomness. Low = focused. High = creative.
0.0 · precise 1.0 · sweet spot 2.0 · chaotic
0.2 (code) · 0.7 (chat) · 1.0 (creative) · >1.5 ⚠ unstable
Max Tokens
Max response length. 1 token ≈ ¾ word.
256 short 1k medium 4k long 128k full doc
More tokens = more cost. Balance wisely.
Top-K
Consider only K most likely next words.
K=5 kept
Range: 1 – vocab size · Typical: 40 – 100
Top-P (nucleus)
Pick tokens whose cumulative probability ≤ P.
42%

6%
cumul: 42 → 68 → 85 → 94% ← nucleus stops here (P=0.9)
Range: 0.0 – 1.0 · Default: 0.9
Pro tip: For code/extraction → Temperature 0.2. For brainstorming → Temperature 0.8–1.0. Avoid changing Top-K and Top-P at the same time.

Workflow
Implications

💾 No Persistent Memory

Each conversation starts fresh. No memory of past sessions unless provided in context.

Workaround: external memory stores, RAG, or conversation summaries

📅 Knowledge Cutoff

Training data has a cutoff date. Models are unaware of events after that date without RAG or tool use.

🎯 Non-Deterministic Output

Probabilistic sampling means same question → different answers. Critical decisions require human oversight.

Tip: set temperature=0 for reproducible outputs in production pipelines

⏱️ Context Window Limits

Models can only process a fixed number of tokens per request (4K–128K+). Long docs must be chunked or handled with RAG.

Design principle: Build around these constraints, not against them. Use RAG for freshness, external memory for persistence, low temperature for consistency, and chunking for long contexts.

What LLMs
Get Wrong

🌀 Hallucinations

LLMs generate plausible-sounding text, not verified facts. They confidently fabricate citations, API names, and statistics.

Real case: a lawyer submitted briefs citing non-existent cases (US, 2023)

🔢 Weak Numerics & Reasoning

LLMs struggle with multi-step arithmetic, logic puzzles, and spatial reasoning. They "feel" the answer rather than compute it.

📦 Invented Libraries & APIs

Models may generate perfectly formatted code using functions that don't exist. Always verify against official docs.

Takeaway: Never trust LLM output blindly — always add verification layers: RAG for facts, code linters for code, human review for critical decisions.

This diagram shows the full architecture of a modern production AI system. At the top, three actors interact with the system: end users who send requests and receive responses, developers who build and maintain the system, and external services that integrate as data sources or consumers. Requests flow down through the application layer — web/mobile front-ends and API backends — into the AI orchestration layer. The orchestration layer is the brain: an agent or planner coordinates everything, calling tools, building prompts, managing context, and routing memory. Below orchestration, three pillars operate in parallel: the model layer (foundation LLM plus an embedding model for similarity search), the tools layer (external APIs, databases, search engines, code execution), and the data and knowledge layer (vector stores, documents, RAG retrieval). A dedicated memory subsystem provides short-term conversation history, long-term per-user memory, and semantic vector-based memory. Underpinning the entire stack is the infrastructure and LLMOps layer: model serving, monitoring, evaluation, logging, safety guardrails, caching, and cost control. Every AI product you use today is some variation of this architecture.

Key Takeaways

Probabilistic, not deterministic

LLMs don't look up answers — they predict the most likely next token. Same prompt, different output. Understand sampling to control behaviour.

Attention is the superpower

The transformer's self-attention mechanism lets every token look at every other token simultaneously. That's why context matters — and why longer context windows are valuable.

Parameters shape behaviour

Temperature controls creativity vs precision. Top-p limits the candidate pool. Context window determines memory. Know these three and you control the model.

No single 'best' model

Choose based on your use case: cost, latency, context length, privacy, and task complexity.

Prompt
Engineering

The best AI in the world gives mediocre results with poor instructions. Master the 4-part anatomy of a prompt and you'll get dramatically better outputs from the same model — starting today.

Chapter 02 · Prompt Engineering

Anatomy of a Prompt

SYSTEM: You are a helpful travel planner. Be friendly and concise. ① Role

USER:
[Context] I'm planning a 5-day trip to Japan in April with my partner. Budget: ~€3,000 total. ② Context
[Task] Create a day-by-day itinerary with must-see places, local food spots, and transport tips. ③ Task
[Format] Use a table: Day | Morning | Afternoon | Evening | Est. Cost. ④ Format
[Constraints] Avoid tourist traps. Include at least one hidden gem per day. ④
  1. 1
    Role / System
    Who is the model? Sets tone, expertise, personality.
  2. 2
    Context
    Background info: who, what, when, budget, constraints.
  3. 3
    Task / Instruction
    What do you want? Be specific and action-oriented.
  4. 4
    Format & Constraints
    Output shape (table, list, JSON), length, rules to follow.

Prompting Patterns

Zero-Shot

No examples given. Direct instruction.

Classify this review as positive or negative:
"The food was amazing."

✓ Best for: simple, well-known tasks

Few-Shot

Provide 2-5 examples to guide the model.

"Great!" → Positive
"Terrible." → Negative
"OK." → Neutral

"The food was amazing." → ?

✓ Best for: custom formats, niche tasks

Chain-of-Thought

Ask the model to reason step by step.

Solve this. Think step by step.

Step 1: identify the variables…
Step 2: apply the formula…

✓ Best for: math, logic, complex reasoning

Rule of thumb: Start with zero-shot. If the output isn't right, add examples (few-shot). For reasoning tasks, always use chain-of-thought.

Advanced Techniques

System Prompts
Hidden instructions that define the model's persona, rules, and guardrails. Users never see them.
SYSTEM: You are a helpful assistant for Acme Corp. Never discuss competitors. Always respond in JSON format.
Role Playing
Assign a specific expert role to improve output quality. The model adapts vocabulary, depth, and style.
Act as a cybersecurity auditor with 15 years of experience…
Output Formatting
Specify exactly what structure you want: JSON, Markdown table, bullet points, YAML, XML…
Return a JSON object with keys: title, summary, score (1-10), tags[]
Meta-Prompting
Ask the model to improve its own prompt. Iterate for better results.
Rewrite this prompt to be clearer, more specific, and get better results…

Do's & Don'ts

✓ Do
  • Be specific and explicit
  • Provide context and examples
  • Specify output format (JSON, table…)
  • Break complex tasks into steps
  • Iterate and refine your prompt
✗ Don't
  • Use vague instructions ("do something good")
  • Send multiple unrelated tasks at once
  • Assume the model remembers past chats
  • Trust output blindly (always verify!)
  • Share sensitive data in prompts

Bad Prompt vs Good Prompt

❌ Vague Prompt
Prompt:
Write something about our product launch.
LLM Output →

"We are happy to announce our new product. It is a great solution that helps users. We hope you enjoy it. Please let us know what you think…"

No audience No format Generic filler
✅ Structured Prompt
Role: You are a B2B marketing copywriter.
Context: We're launching an AI invoice processor that cuts AP workload by 60%.
Task: Write a 3-sentence LinkedIn post announcing the launch.
Format: Hook → key benefit → call to action.
Constraints: No jargon. No exclamation marks. Under 80 words.
LLM Output →

"Finance teams are drowning in manual invoices — we built a way out. Our AI invoice processor cuts accounts payable workload by 60%, integrating with SAP and NetSuite in under a day. See it live at booth 14B, or request a demo at [link]."

Role ✓ Context ✓ Format ✓ Constraints ✓
5 ingredients: Role → Context → Task → Format → Constraints. Each one cuts ambiguity and narrows the output space toward what you actually need.

Real-World Examples

For Managers
Summarize this 20-page report in 5 bullet points. Focus on budget impact and key risks.

→ Zero-shot, format constraint

For Architects
Act as a cloud architect. Compare 3 approaches for real-time order processing: Kafka vs SQS vs EventBridge. Markdown table.

→ Role + format + specificity

For Developers

→ CoT + structured output

RAG Architecture

ChatGPT knows everything that was on the internet before its training cutoff — but nothing about your internal docs, last week's meeting notes, or your proprietary data. RAG is the bridge that changes that.

Chapter 04 · RAG Architecture

Why LLMs Need
External Knowledge

LLM Limitations
  • Knowledge frozen at training cutoff
  • No access to private / enterprise data
  • Can hallucinate outdated facts
  • Context window limits large corpora
RAG Solution
  • Retrieve relevant chunks at query time
  • Ground answers in real, citable sources
  • Works on proprietary documents

Core idea: Instead of retraining the model, inject relevant documents into the prompt at runtime.

Fixed Size

Split every N tokens with a sliding overlap window.

# 512 tok, 10% overlap
"The transformer architecture…"
chunk 1 [0–512]
chunk 2 [460–972] ← overlap

✓ Simple, fast, predictable

✗ May cut sentences mid-thought

Tools

Semantic

Split on sentence / paragraph / heading boundaries.

# Markdown-aware split

→ chunk boundary ↑
## Section: Returns
→ chunk boundary ↑

✓ Preserves meaning, respects structure

✗ Variable sizes, needs structure in source

Hierarchical

Large parent chunks for context + small child chunks for precision.

Parent [2048 tok] ← sent to LLM
├─ Child [128 tok] ← retrieved
├─ Child [128 tok]
└─ Child [128 tok]

✓ High precision retrieval + rich context

✗ More storage, complex pipeline

Rule of thumb: Start with RecursiveCharacterTextSplitter (chunk 512, overlap 64). Switch to Semantic or Hierarchical when retrieval recall drops below 80%.

RAG solves frozen knowledge

Fine-tuning = new skills. RAG = new facts. For enterprise QA, RAG is faster and more explainable.

An LLM answers questions. An AI agent actually does things. This chapter shows how models evolve from passive responders into autonomous workers that plan, use tools, and complete multi-step tasks on your behalf.

Chapter 05 · AI Agents & MCP

What is an AI Agent?

An AI agent is an LLM that can observe its environment, reason, decide, and take actions to reach a goal — in a loop.

LLM vs Agent
Plain LLM
Q → A (one shot)
AI Agent
Goal → Plan → Tools → Result
  1. 1
    Perceive
    Reads input: user message, tool results, memory
  2. 2
    Reason
    Decides what action (or tool) to use next
  3. 3
    Act
    Calls a tool: web search, code exec, API call, file write…
  4. 4
    Observe & Loop
    Checks result, repeats until goal is achieved

Andrew Ng's 4
Agentic Design Patterns

🪞 Reflection
The agent reviews its own output, critiques it, and iterates. Like a writer editing their own draft before submitting.
Example: Code generation → self-review → fix bugs → re-test
🔧 Tool Use
The agent calls external tools — web search, code execution, APIs — to gather information and perform actions beyond text generation.
Example: Research question → search web → read papers → synthesize
📋 Planning
The agent creates an execution plan before acting: breaks complex goals into sub-tasks, orders them, and tracks progress.
Example: "Build a website" → plan → design → code → test → deploy
🤝 Multi-Agent Collaboration
Multiple specialised agents work together: one writes code, another reviews it, a third runs tests. Debate and consensus yield better results.
Example: Writer agent + Critic agent + Editor agent → polished article
Source: Andrew Ng, DeepLearning.AI — "Agentic Design Patterns" series, 2024. These patterns can be combined: an agent can plan, use tools, reflect on results, and collaborate with other agents.

Reason + Act:
The ReAct Loop

Example: "Book me Paris flights for next Tuesday"

Thought: I should confirm before booking.

Observation: "Yes, confirm."

Action: book_flight(flight_id="AF1234")
✓ Booking confirmed. PNR: XK7291

The same technology that summarises medical records can reinforce a doctor's biases. The same agent that books flights can be manipulated to leak your data. Power demands wisdom — this chapter shows how to wield it responsibly.

Chapter 06 · Ethics & Future

Bias & Hallucination

Bias

Models learn patterns from human-generated data — including human prejudices.

  • Gender stereotypes in job descriptions
  • Under-representation of minorities
  • Geographic & cultural bias (Western-centric data)
  • Recency bias from training window
Hallucination

Models generate plausible-sounding but false content with full confidence.

  • Invented citations and papers
  • Wrong legal / medical / technical facts
  • Confident errors in code logic
  • Mitigated by: RAG, grounding, verification
Rule: Never use AI output without human review in high-stakes decisions (hiring, medicine, law, finance).

EU AI Act &
Governance

EU AI Act — In force Aug 2024

The world's first comprehensive AI law. Risk-based approach with four tiers.

🔴 Unacceptable — Prohibited. Social scoring, subliminal manipulation.
🟠 High Risk — Heavy regulation. Healthcare, employment, education, law enforcement.
🟡 Limited Risk — Transparency required. Chatbots must disclose they are AI.
🟢 Minimal Risk — Free to use. Games, spam filters, AI-generated content.

Conclusion

Summary

Ch. 1 · How LLMs Work

LLMs are probabilistic token predictors. They don't "know" facts — they encode statistical patterns. Scale is what unlocked capabilities.

Ch. 2 · Prompt Engineering

How you ask matters as much as what you ask. Role, context, format, and examples dramatically improve output quality.

Ch. 3 · AI-Assisted Coding

AI coding tools excel at boilerplate and tests. They're unreliable for novel algorithms and security-critical code. Always review AI-generated output.

Ch. 4 · RAG Architecture

RAG connects LLMs to private data via vector search, solving the knowledge cutoff problem. Retrieval quality determines answer quality.

Ch. 5 · AI Agents & MCP

Agents combine LLMs with tools and planning for multi-step autonomy. MCP standardises tool connections. Human oversight is essential.

Ch. 6 · Ethics & Future

AI amplifies biases and creates new risks. Responsible AI requires transparency, fairness audits, and human oversight. The technology is neutral — the choices are ours.

Above all: AI is a powerful tool, not magic. Understanding its mechanics, limitations, and risks makes you a better builder, user, and decision-maker.

Key Insights to Remember

🧠
LLMs Are Pattern Machines

They predict next tokens, not truth. Treat every output as a confident draft that needs verification — not a fact.

🎯
Prompts Are Programs

Structured prompts with role, context, constraints, and examples consistently outperform vague questions.

👁️
Always Review AI Output

Whether code, text, or analysis — AI is a co-pilot, not autopilot. The human in the loop is what separates useful AI from dangerous AI.

📚
Context Is King

RAG grounds any model in your data. Retrieval quality determines output quality. Invest in your data pipeline.

🤖
Autonomy Needs Guardrails

AI agents are powerful — and risky. Always scope their authority, validate their actions, and maintain human oversight.

⚖️
Technology Is Neutral — You Are Not

AI amplifies intent. Build with transparency, test for bias, and design for fairness. The choices behind AI matter more than the AI itself.

The Cost of AI

💰 What Drives Cost
📦 Model Size

Larger models cost 10–100× more per call than smaller ones (GPT-4o mini, Haiku).

🔤 Tokens In + Out

Pricing is per token. Long context, verbose prompts all increase cost. Output tokens cost 3–5× more.

🔁 Call Frequency

1M users × 10 msgs/day × 1k tokens = 10B tokens/day → $10k/day at typical pricing.

🗃️ Vector Operations

Embedding + vector search add cost for every RAG query. Caching reduces repeated costs.

🎯 Cost Optimisation Strategies
1. Model routing — use small models for simple tasks

GPT-4o mini for classification, GPT-4o for complex reasoning only.

2. Prompt compression — shorter prompts, same quality

Remove boilerplate, use structured context, define output schema up front.

3. Caching — cache repeated prompts (semantic caching)

GPTCache, Langchain cache: FAQs avoid repeated LLM calls entirely.

4. Fine-tuning — distil GPT-4 behaviour into a 7B model

One-time fine-tuning cost replaces ongoing inference cost for high-volume use cases.

Why run locally?

  • 📶 Offline — no internet connection required
  • 💰 Free — no API costs after setup
  • 🔧 Dev-friendly — custom fine-tunes, no rate limits

# pull & run a model
ollama run llama3.2
ollama run mistral
# OpenAI-compatible API
ollama serve # → localhost:11434
