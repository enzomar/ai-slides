Chapter 04

RAG Architecture

Retrieval-Augmented Generation — grounding language models in your private knowledge.

RAG Architecture
Chapter 4 is critical for anyone building AI products. RAG solves the #1 limitation of LLMs: they don't know your private data and their training has a cutoff. We'll cover the full pipeline — chunking, embedding, vector search, re-ranking, and generation — plus production patterns like hybrid search and evaluation. This is the chapter that turns the audience from AI users into AI builders.
Slide 01 · The Problem

Why LLMs Need
External Knowledge

LLM Limitations
  • Knowledge frozen at training cutoff
  • No access to private / enterprise data
  • Can hallucinate outdated facts
  • Context window limits large corpora
RAG Solution
  • Retrieve relevant chunks at query time
  • Ground answers in real, citable sources
  • Works on proprietary documents
  • Always up-to-date (live index)
Core idea: Instead of retraining the model, inject relevant documents into the prompt at runtime.
The key insight: fine-tuning teaches the model skills and style; RAG teaches it facts. You don't want to fine-tune for facts because they change. Analogy: RAG is like giving the model a search engine + copy-paste rights before it answers. It reads, then responds. This is why RAG is the most common production AI pattern in 2024–2025 — it's cheaper, faster, and more controllable than fine-tuning.

Two phases, two very different timescales. Indexing happens once (or periodically); querying happens every request and must be fast (sub-100ms retrieval). Embedding model choice matters: it must be the same at index time and query time. Cosine similarity is the standard metric — two vectors are similar if they point in the same direction in high-dimensional space.

Slide 03 · Vectors

Embeddings &
Vector Databases

What is an Embedding?

A numerical vector that captures the semantic meaning of text.

"The cat sat on the mat"
→ [0.21, -0.84, 0.13, 0.67, …]
// 1536 dimensions (OpenAI ada-002)

Similar sentences → similar vectors (close in space)

Popular Vector Databases
Pinecone
Managed, cloud-native
Weaviate
Open source, hybrid
Chroma
Local-first, dev-friendly
pgvector
Postgres extension
Qdrant
Rust, high performance
FAISS
Meta, library
Analogy: embeddings are like GPS coordinates for meaning. 'Dog' and 'puppy' have similar coordinates; 'finance' and 'dog' are far apart. The vector DB stores these coordinates and finds nearest neighbours at query time using ANN algorithms (HNSW, IVF). pgvector is the pragmatic enterprise choice — if you already run Postgres, just enable the extension. Pinecone for large-scale managed deployment.

Slide 04 · Chunking

Chunking Strategies

Fixed Size

Split every N tokens (e.g. 512) with overlap.

✓ Simple, predictable

✗ May break sentences mid-thought

Semantic

Split by sentences, paragraphs, or headings.

✓ Preserves meaning

✗ Variable chunk sizes

Hierarchical

Parent chunks for retrieval, child chunks for precision.

✓ Best precision + context

✗ More complex to implement

Rule of thumb: Chunk size ~512 tokens, overlap ~10–15%. Test with your data — no universal best.
The chunking strategy has the largest single impact on RAG quality — even more than the LLM itself. Too small: retrieved chunks lack context. Too large: noisy retrieval, wasted tokens. The overlap prevents straddling important information at chunk boundaries. Advanced: re-ranking with a cross-encoder model after initial retrieval significantly improves relevance (ColBERT, Cohere Rerank).

Slide 05 · Use Cases

RAG in Production

Enterprise Search

Ask questions over internal wikis, SharePoint, Confluence, HR policies.

e.g. Glean, Microsoft 365 Copilot
Support & Docs

Customer-facing chatbots that answer from product docs with source citations.

e.g. Intercom, Zendesk AI
Legal & Compliance

Contract analysis, regulation lookup, policy Q&A with full traceability.

e.g. Harvey, Ironclad AI
Key advantage: Every answer can cite its source. Users and auditors can verify. This is the trust layer AI needs in regulated industries.
RAG is the #1 production AI pattern in 2025. It's cheaper than fine-tuning, safer than raw completions, and auditable. The source citation feature is critical for regulated industries — it transforms a 'black box' into a transparent assistant. Advanced RAG: hybrid search (vector + keyword BM25), re-ranking, query rewriting, HyDE (Hypothetical Document Embeddings).

Slide 06 · Exercise

🛠️ Hands-On Exercise

Challenge: Design a RAG System

Your company wants an internal AI assistant that answers employee questions about HR policies, IT procedures, and company guidelines.

📋 DESIGN DECISIONS
  • What documents go into the knowledge base?
  • How do you chunk a 50-page HR handbook?
  • Which embedding model? (cost vs quality)
  • How do you handle access control (HR Manager vs Intern)?
  • What happens when a document is updated?
⚡ Deliverables
A. Draw the architecture diagram: Data Sources → Chunking → Embedding → Vector DB → Retriever → LLM → Response
B. Choose your tech stack — e.g. OpenAI text-embedding-3-small + Pinecone + GPT-4o-mini
C. Write the system prompt for the assistant (include grounding instructions)
D. Define 3 evaluation queries with expected answers
This is a group exercise (3-4 people, 15-20 minutes). Each group designs the architecture and presents to the class. Common discussion points: chunk size matters — too small loses context, too big dilutes relevance (sweet spot: 500-1000 tokens with 10-20% overlap). Access control: metadata filtering at retrieval time, not post-generation. Document updates: incremental re-indexing pipeline with version tracking. The system prompt should instruct the LLM to say "I don't know" when retrieved context doesn't answer the question — this prevents hallucination in production.
